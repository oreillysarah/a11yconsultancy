Immediate Steps Your Organization Can Take to Prevent an Accessibility Lawsuit

  1. Provide an accessibility statement on your website – The best method to prevent legal action against being inaccessible is having an accessibility statement that outlines your organization’s ongoing commitment to accessibility.
  2. Audit your website – Most accessibility projects begin and end with an audit. They assess the current state of your site or application’s accessibility.
  3. Start simple with free, open-source automated testing tools, such as axe. Such tools can catch between 30-50% of accessibility issues. Axe is also available as a browser extension for Chrome and Firefox, and there are axe tools for testing Android applications and iOS (coming soon) Source (*https://www.deque.com/blog/eu-web-accessibility-compliance-and-legislation/).

Colour contast and low vision

WHO: During the last ten years, cataract (47.9%) remains the leading cause of visual impairment in all areas of the world (except for developed countries). The image below shows how a person with cataracts might see an image on a website. This is why having good colour contrast on your website is essential for website accessibility. People with cataracts or low vision may not be able to use your site and the company will lose out on having them as a customer. WCAG has a specific Web accessibility guideline that deals with colour contrast:

WCAG 1.4.3 Contrast (Minimum)Level AA

The visual presentation of text and images of text has a contrast ratio of at least 4.5:1, except for the following:

  • Large Text: Large-scale text and images of large-scale text have a contrast ratio of at least 3:1;
  • Incidental: Text or images of text that are part of an inactive user interface component, that are pure decoration, that are not visible to anyone, or that are part of a picture that contains significant other visual content, have no contrast requirement.
  • Logotypes: Text that is part of a logo or brand name has no contrast requirement.

Keyboard accessibility, why is it important?

Not everyone uses a standard mouse to interact with a website. Some don’t use a mouse at all! Physical conditions such as cerebral palsy or arthritis (and other physical conditions), can mean that using a standard mouse to operate a computer is impossible. Sometimes an adapted mouse or keyboard is used. This is why keyboard accessibility is very important to consider when designing a website, and why this is a WCAG standard.

WCAG 2.0 Checkpoint 2.1.1 .Keyboard: All functionality of the content is operable through a keyboard interface without requiring specific timings for individual keystrokes, except where the underlying function requires input that depends on the path of the user's movement and not just the endpoints. (Level A)

Coding tips and techniques for accessible headings

To be available to assistive technology, headers must use the following techniques:

  • Ensure each page has a h1 tag and ensure that it is the first item on the page. Furthermore the h1 should have the same name as the page title.
  • Other heading tags used should be in logical order with no levels missing. Nest headings according to their importance e.g. a h2 should not be followed by h5.
  • Headings should be based on the information featured. For example the headings should be the equivalent of a “table of contents” of the key information on a the webpage.

Coding techniques for headings

  • Only use headers for main content structure
  • Do not use text formatting, such as font size or bold to give the visual appearance of headings. For example use
                     <h1>, <h2>, <h2 class=”beta”><h2>

and not

                     <b>, <font>, <span class=”beta”>

Code example for headings:


<html xmlns="http://www.w3.org/1999/xhtml">
<head>
<title>Cooking techniques</title>
</head>
<body>
    <h1>Cooking techniques</h1> ... some text here ...
        <h2>Cooking with oil</h2> ... text of the section ...
            <h3>Sautéeing</h3> ...
            <h3>Deep frying</h3>
        <h2>Cooking with butter</h2> ... text of the section ...
</body>
</html>

Tips for focus indicators for developers and designers

The following WCAG2.0 standards are related to keyboard focus visibility and may overlap:

  • Contrast (Minimum): The visual presentation of text and images of text has a contrast ratio of at least 4.5:1, except for the following: large text, Logotypes and images of text that are part of an inactive user interface component that are pure decoration (WCAG SC 1.4.3).
  • Keyboard Trap - If keyboard focus can be moved to a component of the page using a keyboard interface, then focus can be moved away from that component using only a keyboard interface, and, if it requires more than unmodified arrow or tab keys or other standard exit methods, the user is advised of the method for moving focus away. (WCAG SC 2.1.2)
  • Keyboard Operation – Verify that functionality of the page can be accessed using the tab key and all major features can be completed using only the keyboard. (WCAG SC 1.322)
  • Focus Order – Verify keyboard navigation tab sequence is "visually logical" moving from upper left, proceeding right then wrapping left and down (like reading a book) depending on the elements on the page's rows and column layout. (WCAG SC 2.4.3)

When to use aria labels - coding tips for accessibility

It is better to use native html!! E.g.

input type="radio"

.. As this automatically focuses element. You can use aria for legacy code for custom controls, JS menus, certain JS events. There are times when the

<label>

element falls short—it cannot be used to provide multiple labels for a single form control, or to associate a single label with multiple form controls. There are also times when there is not a visible text label to be associated to a form control (although there should always be visible text where possible!). The majority of these labeling limitations can be overcome with three ARIA properties (aria-labelledby, aria-describedby, and aria-label).
